## Deployment workflow (5 passes)

Always use `make`. The Makefile runs from `terraform/azure/`.

```
Pass 1   make setup-env â†’ make init â†’ make apply          # ~15-20 min
Pass 1.5 make kubeconfig â†’ make k8s-secrets               # cluster access + secrets
Pass 2   make init-values â†’ make deploy                   # Helm deploy (~10 min)
Pass 3   enable_deployments = true   â†’ make init-values && make deploy
Pass 4   enable_agent_builder = true â†’ make init-values && make deploy
Pass 5   enable_insights = true, enable_polly = true â†’ make init-values && make deploy
```

Full status at any point: `make status`

## How the values chain works

`deploy.sh` loads Helm values files in this order (last file wins on conflicts):

```
1. helm/values/values.yaml                              â€” Azure base (NGINX, Blob WI, no Istio)
2. helm/values/values-overrides.yaml                    â€” hostname, WI client-id, auth, postgres/redis
3. helm/values/langsmith-values-sizing-<profile>.yaml  â€” resource requests + HPA settings
4. helm/values/langsmith-values-agent-deploys.yaml     â€” Pass 3 (if enable_deployments)
5. helm/values/langsmith-values-agent-builder.yaml     â€” Pass 4 (if enable_agent_builder)
6. helm/values/langsmith-values-insights.yaml          â€” Pass 5 (if enable_insights)
7. helm/values/langsmith-values-polly.yaml             â€” Pass 5 (if enable_polly)
```

Files 3â€“7 are copies of `examples/` files, generated by `init-values.sh`. The examples are the source of truth â€” edit examples, then re-run `make init-values` to propagate.

**Do not set `replicaCount` or `resources` for shared components (backend, frontend, playground, aceBackend) in addon files.** Those components are sized by the sizing file which loads before addons. Setting them in an addon file silently overrides HPA settings.

## Key files to understand before editing

### `infra/scripts/_common.sh`
Sourced by every script. Two critical helpers:

```bash
_parse_tfvar "key"        # reads quoted or unquoted value from terraform.tfvars
_tfvar_is_true "key"      # returns 0 if key = true or "true"
```

`_parse_tfvar` handles both `key = "string"` and `key = true` / `key = 42`. The two-step sed pattern is intentional â€” quoted strings use a capture group, unquoted values strip the assignment prefix. Do not simplify to a single sed expression; it was fixed after a bug where unquoted booleans silently fell through and init-values.sh never generated addon files.

### `infra/scripts/setup-env.sh`
**Read-only against Key Vault.** It reads from KV but never writes. Terraform is the sole KV writer. This prevents state drift during `terraform apply`. Stable secret priority: Key Vault â†’ local dot-file fallback â†’ generate fresh.

### `helm/scripts/init-values.sh`
Reads `terraform.tfvars` and `terraform output`, writes `values-overrides.yaml` and copies/generates addon files. Hostname resolution priority:
1. `langsmith_domain` (custom domain)
2. `dns_label` â†’ `<label>.<region>.cloudapp.azure.com` (works for all ingress controllers)
3. Existing value in `values-overrides.yaml` (preserved on re-run)
4. Interactive prompt

For insights with `clickhouse_source = "in-cluster"`, generates a minimal file (`config.insights.enabled: true` only) instead of copying the external ClickHouse example. This is intentional â€” the external example sets `clickhouse.external.enabled: true` and `existingSecretName: langsmith-clickhouse`, which causes `CreateContainerConfigError` when the secret doesn't exist.

### `helm/scripts/deploy.sh`
Does more than `helm upgrade`. Key non-obvious steps:
- Annotates the ingress LoadBalancer service with `service.beta.kubernetes.io/azure-dns-label-name` â€” targets the correct service based on `ingress_controller` (nginx/istio/istio-addon/agic/envoy-gateway). Without this annotation, Azure never assigns the DNS label to the public IP and cert-manager's HTTP-01 challenge fails. AGIC uses the AGW public IP FQDN instead of a K8s annotation
- Creates `letsencrypt-prod` ClusterIssuer if `tls_certificate_source = "letsencrypt"` â€” cannot be done in Terraform because `kubernetes_manifest` requires a live API server during `terraform plan`, which doesn't exist on a fresh deploy
- Auto-rolls back a `pending-upgrade` Helm release state before proceeding
- Annotates `langsmith-ksa` service account with Workload Identity client-id â€” this SA is used by operator-spawned agent deployment pods

## terraform.tfvars fields â€” what matters


## Workload Identity (WI) â€” how it works

AKS OIDC issuer + Azure managed identity + federated credentials = pods access Azure Blob without static credentials.

The managed identity is created in `k8s-cluster` module. Federated credentials for each service account are registered there. To add a new pod that needs Blob access: update `service_accounts_for_workload_identity` in `modules/k8s-cluster/variables.tf` and run `terraform apply -target=module.aks`.

Pods need two things:
1. Pod label: `azure.workload.identity/use: "true"`
2. Service account annotation: `azure.workload.identity/client-id: <managed-identity-client-id>`

Both are set in `values-overrides.yaml` by `init-values.sh` for the 5 blob-accessing service accounts. `langsmith-ksa` (used by operator-spawned agent pods) gets its annotation applied by `deploy.sh` after Helm install.

## Known gotchas (validated during full E2E test â€” all 5 passes)

**DNS label not resolving / cert stuck Pending**
NGINX LoadBalancer needs `service.beta.kubernetes.io/azure-dns-label-name` annotation. `deploy.sh` sets this. If deploying manually: `kubectl annotate svc ingress-nginx-controller -n ingress-nginx service.beta.kubernetes.io/azure-dns-label-name=<label> --overwrite`

**`letsencrypt-prod` ClusterIssuer missing**
`deploy.sh` creates it idempotently. If cert is stuck Pending: `kubectl describe certificate -n langsmith` â€” look for "issuer not found". Re-run `make deploy` to recreate it, or apply the inline manifest from `deploy.sh` manually.

**`enable_deployments = true` not taking effect**
`_parse_tfvar` must handle unquoted booleans. Current implementation has two-step sed (quoted + unquoted fallback). If broken, the old single-group sed `s/.*=[[:space:]]*"\([^"]*\)".*/\1/` returns empty for unquoted values and init-values.sh never generates addon files.

**Deployments stuck in DEPLOYING state**
`config.deployment.url` is empty or `config.deployment.tlsEnabled` is wrong. `init-values.sh` injects these via python3 substitution. Verify: `grep -E 'url:|tlsEnabled' helm/values/langsmith-values-agent-deploys.yaml`

**Pass 5 `CreateContainerConfigError` â€” `langsmith-clickhouse` secret not found**
Happens when `langsmith-values-insights.yaml` was manually copied from `examples/` for an in-cluster ClickHouse deploy. The example has `clickhouse.external.enabled: true` and `existingSecretName: langsmith-clickhouse`. Fix: `make init-values` regenerates the correct minimal file.

**`make destroy` hangs**
Caused by active LoadBalancer with public IP. Run `make uninstall` first. If already stuck: `az group delete --name langsmith-rg-<identifier> --yes --no-wait`

**`make uninstall` hangs on namespace deletion**
The prompt times out in non-interactive shells. Run `kubectl delete namespace langsmith --grace-period=0 --force`

**`agent_builder requires deployments` error**
`enable_agent_builder = true` requires `enable_deployments = true`. Both must be set.

**AGIC pod CrashLoopBackOff â€” persistent 403 on Application Gateway**
Root cause: AKS creates the AGIC addon managed identity during cluster provisioning, but Azure AD needs ~5 minutes to register the service principal before role assignments become effective. Terraform now adds a `time_sleep` (300s) before AGIC role assignments to prevent this on fresh deploys. For existing clusters already stuck: `az aks update --name <cluster> --resource-group <rg> --yes` then `kubectl delete pod -n kube-system -l app=ingress-azure`. See INGRESS_CONTROLLERS.md for full details.

**Terraform "Invalid count argument" on AGIC role assignments**
Happens if `count` on AGIC role assignments references an apply-time local (e.g. `local.agic_addon_principal_id != null`). Must use plan-time variable only: `var.ingress_controller == "agic" ? 1 : 0`. This is already fixed in `modules/k8s-cluster/main.tf`.

## What NOT to do

- **Do not run `terraform apply` directly.** Use `make apply` â€” it ensures `secrets.auto.tfvars` exists first.
- **Do not run `helm upgrade` directly.** `deploy.sh` handles DNS annotation, ClusterIssuer, and values chain. Manual helm skips all of that.
- **Do not commit `terraform.tfvars` or `secrets.auto.tfvars`.** Both are gitignored. They contain subscription IDs, passwords, and license keys.
- **Do not set secrets inline in values files.** Secrets flow through `langsmith-config-secret`. Inline secrets appear in Helm release metadata and `helm get values` output.
- **Do not use `kubernetes_manifest` in Terraform for resources that need a live cluster.** It requires the k8s API during `terraform plan` â€” fails on a fresh deploy when no cluster exists yet. Use `kubectl apply` in `deploy.sh` instead.
- **Do not add `resources` or `replicaCount` for shared components in addon files.** The sizing file loads before addons and owns those fields. An addon override silently wins the merge.
- **Do not edit generated files directly** (`values-overrides.yaml`, `langsmith-values-sizing-*.yaml`, `langsmith-values-agent-*.yaml`). They are overwritten by `make init-values`. Edit `examples/` instead.
