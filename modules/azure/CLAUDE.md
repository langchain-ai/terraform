# CLAUDE.md — terraform/azure/

This is the Azure LangSmith deployment directory. It provisions AKS + managed services via Terraform and deploys LangSmith via Helm. Everything is driven through `make` — do not run `terraform`, `helm`, or `kubectl` directly unless debugging a specific issue.

## Deployment workflow (5 passes)

Always use `make`. The Makefile runs from `terraform/azure/`.

```
Pass 1   make setup-env → make init → make apply          # ~15-20 min
Pass 1.5 make kubeconfig → make k8s-secrets               # cluster access + secrets
Pass 2   make init-values → make deploy                   # Helm deploy (~10 min)
Pass 3   enable_deployments = true   → make init-values && make deploy
Pass 4   enable_agent_builder = true → make init-values && make deploy
Pass 5   enable_insights = true, enable_polly = true → make init-values && make deploy
```

Full status at any point: `make status`

## How the values chain works

`deploy.sh` loads Helm values files in this order (last file wins on conflicts):

```
1. helm/values/values.yaml                              — Azure base (NGINX, Blob WI, no Istio)
2. helm/values/values-overrides.yaml                    — hostname, WI client-id, auth, postgres/redis
3. helm/values/langsmith-values-sizing-<profile>.yaml  — resource requests + HPA settings
4. helm/values/langsmith-values-agent-deploys.yaml     — Pass 3 (if enable_deployments)
5. helm/values/langsmith-values-agent-builder.yaml     — Pass 4 (if enable_agent_builder)
6. helm/values/langsmith-values-insights.yaml          — Pass 5 (if enable_insights)
7. helm/values/langsmith-values-polly.yaml             — Pass 5 (if enable_polly)
```

Files 3–7 are copies of `examples/` files, generated by `init-values.sh`. The examples are the source of truth — edit examples, then re-run `make init-values` to propagate.

**Do not set `replicaCount` or `resources` for shared components (backend, frontend, playground, aceBackend) in addon files.** Those components are sized by the sizing file which loads before addons. Setting them in an addon file silently overrides HPA settings.

## Key files to understand before editing

### `infra/scripts/_common.sh`
Sourced by every script. Two critical helpers:

```bash
_parse_tfvar "key"        # reads quoted or unquoted value from terraform.tfvars
_tfvar_is_true "key"      # returns 0 if key = true or "true"
```

`_parse_tfvar` handles both `key = "string"` and `key = true` / `key = 42`. The two-step sed pattern is intentional — quoted strings use a capture group, unquoted values strip the assignment prefix. Do not simplify to a single sed expression; it was fixed after a bug where unquoted booleans silently fell through and init-values.sh never generated addon files.

### `infra/scripts/setup-env.sh`
**Read-only against Key Vault.** It reads from KV but never writes. Terraform is the sole KV writer. This prevents state drift during `terraform apply`. Stable secret priority: Key Vault → local dot-file fallback → generate fresh.

### `helm/scripts/init-values.sh`
Reads `terraform.tfvars` and `terraform output`, writes `values-overrides.yaml` and copies/generates addon files. Hostname resolution priority:
1. `langsmith_domain` (custom domain)
2. `nginx_dns_label` → `<label>.<region>.cloudapp.azure.com`
3. `frontdoor_endpoint_hostname` from terraform output
4. Existing value in `values-overrides.yaml` (preserved on re-run)
5. Interactive prompt

For insights with `clickhouse_source = "in-cluster"`, generates a minimal file (`config.insights.enabled: true` only) instead of copying the external ClickHouse example. This is intentional — the external example sets `clickhouse.external.enabled: true` and `existingSecretName: langsmith-clickhouse`, which causes `CreateContainerConfigError` when the secret doesn't exist.

### `helm/scripts/deploy.sh`
Does more than `helm upgrade`. Key non-obvious steps:
- Annotates the NGINX LoadBalancer service with `service.beta.kubernetes.io/azure-dns-label-name` — without this, Azure never assigns the DNS label to the public IP and cert-manager's HTTP-01 challenge fails
- Creates `letsencrypt-prod` ClusterIssuer if `tls_certificate_source = "letsencrypt"` — cannot be done in Terraform because `kubernetes_manifest` requires a live API server during `terraform plan`, which doesn't exist on a fresh deploy
- Auto-rolls back a `pending-upgrade` Helm release state before proceeding
- Annotates `langsmith-ksa` service account with Workload Identity client-id — this SA is used by operator-spawned agent deployment pods

## terraform.tfvars fields — what matters

```hcl
identifier      = "-azonf"          # suffix on ALL resource names (rg, aks, kv, blob...)
                                    # changing this creates entirely new resources

postgres_source   = "in-cluster"    # "in-cluster" | "external"
redis_source      = "in-cluster"    # "in-cluster" | "external"
clickhouse_source = "in-cluster"    # "in-cluster" | "external" (in-cluster = dev/POC only)

nginx_dns_label        = "langsmith-azonf"    # → langsmith-azonf.eastus.cloudapp.azure.com
tls_certificate_source = "letsencrypt"        # "letsencrypt" | "dns01" | "none"
letsencrypt_email      = "you@example.com"    # used in ClusterIssuer ACME config

sizing_profile = "minimum"          # minimum | dev | production | production-large

# Pass 3-5 flags — change these, then run: make init-values && make deploy
# No terraform apply needed for these — they only affect Helm values
enable_deployments   = true
enable_agent_builder = true
enable_insights      = true
enable_polly         = true
```

`keyvault_purge_protection = false` is the dev/test default. On destroy the KV name is freed immediately via `az keyvault purge`. In production set to `true` — but then you cannot reuse the same `identifier` for 90 days after destroy.

## Secrets architecture

```
setup-env.sh prompts
      ↓
secrets.auto.tfvars  (gitignored, picked up automatically by terraform)
      ↓
terraform apply  →  Azure Key Vault  (Terraform is the sole KV writer)
      ↓
make k8s-secrets  →  langsmith-config-secret  (K8s Secret in langsmith namespace)
      ↓
Helm chart reads via config.existingSecretName: "langsmith-config-secret"
```

The `langsmith-config-secret` holds 8 keys: `api_key_salt`, `jwt_secret`, `langsmith_license_key`, `initial_org_admin_password`, `deployments_encryption_key`, `agent_builder_encryption_key`, `insights_encryption_key`, `polly_encryption_key`.

Two additional secrets are created by Terraform (k8s-bootstrap module), not by `make k8s-secrets`:
- `langsmith-postgres-secret` — `connection_url` key
- `langsmith-redis-secret` — `connection_url` key

## Workload Identity (WI) — how it works

AKS OIDC issuer + Azure managed identity + federated credentials = pods access Azure Blob without static credentials.

The managed identity is created in `k8s-cluster` module. Federated credentials for each service account are registered there. To add a new pod that needs Blob access: update `service_accounts_for_workload_identity` in `modules/k8s-cluster/variables.tf` and run `terraform apply -target=module.aks`.

Pods need two things:
1. Pod label: `azure.workload.identity/use: "true"`
2. Service account annotation: `azure.workload.identity/client-id: <managed-identity-client-id>`

Both are set in `values-overrides.yaml` by `init-values.sh` for the 5 blob-accessing service accounts. `langsmith-ksa` (used by operator-spawned agent pods) gets its annotation applied by `deploy.sh` after Helm install.

## Known gotchas (validated during full E2E test — all 5 passes)

**DNS label not resolving / cert stuck Pending**
NGINX LoadBalancer needs `service.beta.kubernetes.io/azure-dns-label-name` annotation. `deploy.sh` sets this. If deploying manually: `kubectl annotate svc ingress-nginx-controller -n ingress-nginx service.beta.kubernetes.io/azure-dns-label-name=<label> --overwrite`

**`letsencrypt-prod` ClusterIssuer missing**
`deploy.sh` creates it idempotently. If cert is stuck Pending: `kubectl describe certificate -n langsmith` — look for "issuer not found". Manual fix: `kubectl apply -f kubectl/letsencrypt-issuers.yaml`

**`enable_deployments = true` not taking effect**
`_parse_tfvar` must handle unquoted booleans. Current implementation has two-step sed (quoted + unquoted fallback). If broken, the old single-group sed `s/.*=[[:space:]]*"\([^"]*\)".*/\1/` returns empty for unquoted values and init-values.sh never generates addon files.

**Deployments stuck in DEPLOYING state**
`config.deployment.url` is empty or `config.deployment.tlsEnabled` is wrong. `init-values.sh` injects these via python3 substitution. Verify: `grep -E 'url:|tlsEnabled' helm/values/langsmith-values-agent-deploys.yaml`

**Pass 5 `CreateContainerConfigError` — `langsmith-clickhouse` secret not found**
Happens when `langsmith-values-insights.yaml` was manually copied from `examples/` for an in-cluster ClickHouse deploy. The example has `clickhouse.external.enabled: true` and `existingSecretName: langsmith-clickhouse`. Fix: `make init-values` regenerates the correct minimal file.

**`make destroy` hangs**
Caused by active LoadBalancer with public IP. Run `make uninstall` first. If already stuck: `az group delete --name langsmith-rg-<identifier> --yes --no-wait`

**`make uninstall` hangs on namespace deletion**
The prompt times out in non-interactive shells. Run `kubectl delete namespace langsmith --grace-period=0 --force`

**`agent_builder requires deployments` error**
`enable_agent_builder = true` requires `enable_deployments = true`. Both must be set.

## Teardown sequence

```bash
make uninstall   # removes Helm releases, LGP CRD, namespaces
make destroy     # destroys all Azure infrastructure
make clean       # removes local secrets, generated values files
# then purge KV if keyvault_purge_protection = false:
az keyvault purge --name "langsmith-kv-<identifier>" --location eastus
```

## Adding a new addon (Pass 6+)

1. Add `enable_<addon>` variable to `infra/variables.tf` and `infra/main.tf`
2. Create `helm/values/examples/langsmith-values-<addon>.yaml`
3. Add to `init-values.sh` — read the flag with `_parse_tfvar`, call `_copy_addon` if true
4. Add to `deploy.sh` `_addon_gate` array in the same order as init-values
5. Update `README.md` Deployment Passes table and Helm Values Files section
6. Test: set the flag in `terraform.tfvars`, run `make init-values && make deploy`, verify `make status`

## What NOT to do

- **Do not run `terraform apply` directly.** Use `make apply` — it ensures `secrets.auto.tfvars` exists first.
- **Do not run `helm upgrade` directly.** `deploy.sh` handles DNS annotation, ClusterIssuer, and values chain. Manual helm skips all of that.
- **Do not commit `terraform.tfvars` or `secrets.auto.tfvars`.** Both are gitignored. They contain subscription IDs, passwords, and license keys.
- **Do not set secrets inline in values files.** Secrets flow through `langsmith-config-secret`. Inline secrets appear in Helm release metadata and `helm get values` output.
- **Do not use `kubernetes_manifest` in Terraform for resources that need a live cluster.** It requires the k8s API during `terraform plan` — fails on a fresh deploy when no cluster exists yet. Use `kubectl apply` in `deploy.sh` instead.
- **Do not add `resources` or `replicaCount` for shared components in addon files.** The sizing file loads before addons and owns those fields. An addon override silently wins the merge.
- **Do not edit generated files directly** (`values-overrides.yaml`, `langsmith-values-sizing-*.yaml`, `langsmith-values-agent-*.yaml`). They are overwritten by `make init-values`. Edit `examples/` instead.
